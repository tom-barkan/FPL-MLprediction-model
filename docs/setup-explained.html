What We Just Built (And Why)

A plain-English guide to everything we set up for the FPL prediction project

The Big Picture

We're building a project that predicts how many Fantasy Premier League points a player will score each week. We'll use two different AI approaches and compare which one is better.

But before we can write any of that clever code, we need to set up our workshop. That's what this page covers — everything we did today was preparation, not the actual building.

Think of it like cooking: Before you cook a meal, you need a kitchen (your computer), the right appliances (software tools), ingredients in the pantry (data), and recipe books (code libraries). Today we installed the appliances and organised the pantry shelves.
1

Installed Python 3.11

The programming language everything runs on

Python is the language we'll write all our code in. Your Mac already had an older version (3.9), but some of the tools we need require a newer one. So we installed version 3.11.

Real-world analogy: Your Mac had an old blender that couldn't crush ice. We bought a newer blender that has all the features we need. The old one is still there — we just use the new one for this project.
What we ran
brew install python@3.11

Homebrew is like an app store for developer tools on Mac. Instead of downloading installers from websites, you just type one command and it handles everything.

2

Created a Virtual Environment

A private sandbox so our project doesn't mess up other things

A virtual environment is a sealed-off space where we install tools only for this project. Without it, installing something for this project could accidentally break something else on your computer.

Real-world analogy: Imagine you have one big tool shed shared by every project in your house. If you rearrange it for a plumbing job, you might lose track of the tools you need for the electrical job. A virtual environment is like giving each project its own personal tool shed.
What we ran
python3.11 -m venv .venv

This created a folder called .venv inside the project. The dot at the front means it's hidden — it's there, but it stays out of the way. Every time we work on this project, we "activate" this environment so the computer knows to use our tools, not the system-wide ones.

3

Installed All the Tools We Need

14 software packages, each with a specific job

We created a shopping list called requirements.txt and installed everything on it. Here's what each tool does in plain English:

  • requests — Lets our code talk to the FPL website and download player data
  • pandas — A spreadsheet engine for Python. Lets us sort, filter, and organise data like you would in Excel, but with code
  • scikit-learn — A toolkit of standard AI/statistics methods. We use it to measure how good our predictions are
  • xgboost — Our first prediction model. It's a well-proven method that learns patterns from tables of numbers (like a really smart spreadsheet formula)
  • mlx — Apple's own AI engine, designed to run fast on your Mac's M4 chip
  • mlx-lm — Lets us fine-tune (customise) a language model using MLX. This is how we'll teach an AI to predict FPL points
  • huggingface_hub — Connects to Hugging Face, a website where people share pre-built AI models. We'll download our starting model from there
  • streamlit — Turns Python code into a web dashboard with almost no effort. We'll use it to compare our two models visually
  • matplotlib — Creates charts and graphs from data
  • jupyter / nbformat / nbconvert — Jupyter notebooks let you mix code, charts, and written explanations in one document. Great for showing your work
  • pyyaml — Reads configuration files (our evaluation test definitions)
  • psutil — Measures how much memory and time our models use

We also had to install libomp (a helper that XGBoost needs to do its calculations quickly on Mac).

What we ran
pip install -r requirements.txt
brew install libomp
4

Created the Folder Structure

Organised empty folders so everything has a home

Before writing any code, we set up folders so every file we create has an obvious place to go. It's like labelling drawers before you start unpacking boxes.

data/ — All the FPL player data
    raw/ — Data straight from the FPL website (unprocessed)
    processed/ — Data cleaned up and ready for the models
    labelled/ — Data you'll manually annotate yourself
    mlx/ — Data formatted for the language model
models/ — Training scripts and saved models
eval/ — Tests to measure how good the models are
    cases/ — Specific test scenarios
    checks/ — Quality validators
    results/ — Scores from each experiment
    baseline/ — The "best so far" to compare against
notebooks/ — Jupyter notebooks documenting each phase
ui/ — The comparison dashboard
writeup/ — Final assessment and product brief
docs/ — Published web pages (like this one!)
5

Created a .gitignore File

A list of things we don't want to share online

Git tracks every change you make to your code and lets you share it on GitHub. But some things shouldn't be shared:

  • AI model files — They're huge (2GB+) and can be re-downloaded
  • Raw data — Can be re-fetched from the FPL website anytime
  • The virtual environment — Anyone can recreate it from requirements.txt
  • System junk files — Things like .DS_Store that macOS creates automatically
Real-world analogy: When you move house, you pack your personal belongings but you don't pack the tap water or the electricity — those are available at the new address. .gitignore is the list of things that don't need to travel with the project.
6

Created requirements.txt

The recipe card that lists all ingredients

This is a simple text file listing every tool we installed. If you (or anyone) ever needs to set this project up on a different computer, they just run one command and everything gets installed automatically.

Real-world analogy: It's the ingredients list at the top of a recipe. You don't need to figure out what you need — just take the list to the shop.

Jargon Buster

Homebrew
A free tool for Mac that lets you install software by typing commands instead of downloading from websites. Like an app store for developer tools.
Virtual Environment (venv)
A private, isolated copy of Python and its tools for one specific project. Stops different projects from interfering with each other.
pip
Python's built-in tool for installing packages (add-on tools). Think of it as "please install package."
Git / GitHub
Git tracks every change you make to your code (like "track changes" in Word). GitHub is a website where you can share that tracked code with others.
.gitignore
A list of files and folders that Git should pretend don't exist. Used for big files, temporary files, and private data.
XGBoost
A popular AI method that's great at making predictions from tables of numbers. It learns patterns like "players in good form at home against weak teams score more."
LLM (Large Language Model)
An AI that understands and generates human language (like ChatGPT). We're going to customise one to predict FPL points from player descriptions.
Fine-tuning
Taking a pre-built AI model and teaching it something new with your own data. Like hiring a smart employee and training them on your specific business.
MLX
Software from Apple that lets AI models run efficiently on Mac's M-series chips. Without it, you'd normally need an expensive cloud GPU.
LoRA
A technique for fine-tuning AI models efficiently. Instead of retraining the whole model (expensive), it makes small, targeted adjustments. Like hemming trousers instead of sewing a new pair.
API
A way for software to request data from a website. The FPL API lets our code ask "give me all player stats" and get a structured answer back.
